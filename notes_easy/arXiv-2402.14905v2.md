# MobileLLM: スマホで動く「小さくて賢い」言語AIの作り方（日本語パラフレーズ）

- arXiv: https://arxiv.org/abs/2402.14905
- 一次ソース: ../papers/arXiv-2402.14905v2/
- 正規ノート: ../notes/arXiv-2402.14905v2.md

## 一言で言うと

スマホでも動くサイズ（10億パラメータ未満）の言語AIは、データ量より「形（設計）」をうまく工夫するだけで、ぐっと賢くなるよ、という論文。

## どんな問題を解こうとしてるの？

- 最近の言語AI（ChatGPT みたいなやつ）は、クラウド（インターネットの向こうにある大きなコンピュータ）で動いているけど、すごく電気を食うし、お金もかかるし、返事が来るまで時間もかかる。
- スマホで直接動かせれば、ネットに繋がなくても返事が来るし、個人の話も外に出ない。でも、スマホは「メモリ（DRAM、データを一時的に置く部屋）」がとても狭い。
  - iPhone 15 で 6GB、Pixel 8 Pro で 12GB しかなくて、他のアプリと共有なので、AI は **DRAM の 10% くらい** しか使えない。
  - また、電池の問題もあって、70 億パラメータの大きい AI を動かすと、満タンの iPhone が **2 時間もたない**（論文の試算より）。
- だから「10 億パラメータ未満」の小さい AI を本気で良くしたい。なのに今までの小さい AI（OPT-125M、Pythia-160M、GPT-Neo-125M など）は、大きいモデルの「ミニチュア版」みたいな作り方で、小さいサイズ専用に最適化されてなかった。
- さらに、過去には「AI の賢さはデータ量とパラメータ数で決まり、形はあまり関係ない」という有名な主張（**scaling law**、後述）があって、形を工夫する研究があまり進んでいなかった。

身近な例えで言うと、お弁当箱に料理を詰めるとき、「容器が小さい時こそ、おかずの並べ方を工夫しないと入らない」よね。この論文は「小さい弁当箱の詰め方」を真面目に研究したもの。

## どうやって解いたの？

### コアアイデア

著者のひらめきは、シンプルに言うと「**小さい AI は『太く短い』より『細く長い』ほうが賢くなる**」「**同じ部品を使い回す**と、サイズを増やさず賢くできる」の 2 つ。

例え話：レゴで何かを作るとき、ブロックの個数が決まっているとする。広く 1 段だけ並べるより、細くても高く積み上げた方が、複雑な形（=複雑な考え方）が表現できる。MobileLLM はこれを言語 AI の「層（layer）」の数で実現した。さらに「**同じレゴブロックを 2 回使って数える**」みたいなトリック（layer sharing）で、見かけの部品数を増やさず段を増やす。

### 仕組み

論文では、強い小型 AI「MobileLLM」を 4 つの工夫で作り、さらに 5 つ目の工夫を足した「MobileLLM-LS」を提案している。

- **step1: 部品「FFN」を SwiGLU 化** — AI の中で計算する部品の 1 つ（後述）を、新しいタイプに変えた。これだけで 125M モデルの平均点が **42.6 → 43.9** に上がった。
- **step2: 深く・細く (deep-and-thin)** — 層の数（building block の段数）を増やして、1 段あたりの幅は減らす。論文では 19 個ものモデルを実際に学習して比べ、「**125M なら 30〜42 層、350M なら 32 層** が良い」と結論。今までの 125M モデルは 12 層が普通だったので大胆な変更。
- **step3: 入口と出口で同じ「単語表」を使い回す (embedding sharing)** — 「単語 → 数の並び」に変える表と、「数の並び → 単語の確率」に戻す表を共有する。125M モデルだと **約 1,600 万パラメータ（全体の約 11.8%）が節約できる**。浮いた分を層の追加に回す。
- **step4: grouped query attention (GQA)** — Attention（後述）で使う部品のうち、「key/value」と呼ばれる部品の数を「query」の 1/n に減らして共有する。125M で精度がさらに +0.4 上がった。
- **step5: ブロック単位の重み共有 (immediate block-wise layer sharing)** — 隣り合う 2 つのブロックで同じ重みを使う（=同じブロックを 2 回連続で計算する）。新しいパラメータは増えないが層が増えた効果が出て、精度が **+0.7 〜 +0.8** 上がる。さらに、スマホの高速メモリ「SRAM（後述）」にちょうど 1 ブロック分が乗るので、メモリの出し入れ時間が節約できて速度もほぼ落ちない。

### 主要な数式

この論文は新しい数式を提案する論文ではないけれど、「中で何をやっているのか」を理解する上で大事な計算が 2 つあるので、それを噛み砕いて紹介するよ。

#### (1) Attention（注目）の計算

言語 AI の中心にある計算。「文の中のどの単語にどれくらい注目するか」を決める式。

$$ \text{Attention}(Q, K, V) = \text{softmax}\!\left( \frac{Q K^\top}{\sqrt{d_k}} \right) V $$

**この式が言ってること**: 入力された各単語について「他のどの単語をどれくらい見るか」のスコアを計算して、見るべき単語の情報を重み付き平均して取り出している。「みんなで投票して、票が多い人の意見を強く採用する」イメージ。

**記号の意味**:
- $Q$ … query（質問役）。「自分が今、何を知りたいか」をあらわす数の並び。
- $K$ … key（看板役）。「自分はこういう情報を持ってるよ」をあらわす数の並び。
- $V$ … value（中身）。実際に取り出される情報の数の並び。
- $Q K^\top$ … 質問と看板の「相性スコア」を全ペア分計算したもの（内積。数の並び同士の「向きが似てる度」を測る計算）。
- $d_k$ … key の長さ（数の並びがいくつあるか）。$\sqrt{d_k}$ で割るのは、スコアが大きくなりすぎないようにするおまじない。
- softmax … スコアを「合計が 1 になる比率」に整える関数。たとえば (2, 1, 1) → (0.5, 0.25, 0.25) のように、大きいほど比率が大きくなる。

**身近な例え**: クラスでグループ発表のテーマを決める場面。あなた（$Q$）が「恐竜の話をしたい」と思っているとする。クラスメイト全員が「私は宇宙が得意」「私は生き物が得意」と看板（$K$）を持っている。看板と相性が良い人ほど大きな声で意見($V$)を言ってもらい、それを集めて自分の最終アイデアにする。MobileLLM では、この「看板」と「中身」をクラス全員ぶん持たずに、何人かで共有する（GQA）ことで節約している。

#### (2) Grouped Query Attention のおおまかな式

GQA は (1) の Attention の変形版。query は head ごとに別々に用意するけど、key と value は何個かで 1 個を共有する。

$$ \text{head}_i = \text{Attention}\!\left( Q_i,\; K_{\lfloor i/g \rfloor},\; V_{\lfloor i/g \rfloor} \right) $$

**この式が言ってること**: $i$ 番目の head（注目のグループ）は、自分専用の query $Q_i$ を持つけれど、key と value は「$g$ 個まとめて 1 セット」しかなくて、近所同士で同じものを使い回す、ということ。

**記号の意味**:
- $i$ … 何番目の head か（0, 1, 2, … と番号がついている）。
- $g$ … いくつの query head で 1 セットの key/value を共有するか（論文の 125M モデルだと query 9 個・kv 3 個なので $g=3$）。
- $\lfloor i/g \rfloor$ … $i$ を $g$ で割った答えの整数部分（4÷3 なら 1）。これで「どのグループに属するか」が決まる。
- $Q_i, K, V$ … 上の (1) と同じ意味の質問・看板・中身。

**身近な例え**: 学校で班を作る場面。9 人の生徒（query）がいて、ペンセット（key/value）は 3 セットしかない。1〜3 番が 1 セット目を共有、4〜6 番が 2 セット目、7〜9 番が 3 セット目を共有する。ペンセットの数は減るけど、生徒 1 人 1 人の発想は別々なので、思ったほど結果は悪くならない。

#### (3) Block-wise Layer Sharing（ブロックの使い回し）

これは式というより「定義」だけど、論文の中心アイデアなので書いておく。

$$ h_{l+1} = f_{\theta_{l}}(h_l), \qquad h_{l+2} = f_{\theta_{l}}(h_{l+1}) $$

**この式が言ってること**: $l$ 番目のブロックと $l{+}1$ 番目のブロックで「同じ重み $\theta_l$ を使い回す」。つまり同じブロックを 2 回連続で適用する。重みは 1 個ぶんしか持っていないのに、見かけの層の深さは 2 倍。

**記号の意味**:
- $h_l$ … $l$ 段目を通過した後の「途中の表現」（数の並び。文の理解度合いのようなもの）。
- $f_{\theta_l}$ … $l$ 段目のブロックがやる計算（中身は Attention や FFN）。$\theta_l$ はそのブロックの重み（覚えた知識）。
- $h_{l+1}$ … 1 回通したあとの状態。これをもう 1 回同じブロックに通すと $h_{l+2}$ が出てくる。

**身近な例え**: お料理で「煮込む」工程を 2 回繰り返すと、味がしみてもっと美味しくなるイメージ。同じ鍋（=重み）を使うので新しい道具は要らないけど、時間（計算）は 2 倍。論文ではこれが「精度を上げる魔法」になる、と発見した。

## 何がすごいの？

論文に書かれている結果を、TeX から拾って書き出すよ。

- **ベンチマーク（学力テスト）の平均点（zero-shot common sense reasoning）**
  - MobileLLM-125M: **46.3** / MobileLLM-LS-125M: **47.0**
    - これまでの 125M〜170M 帯のベスト RWKV-169M は 43.6、Pythia-160M は 42.5。**+2.7〜+3.8 ポイント**勝った上に、サイズは **22〜26% 小さい**。
  - MobileLLM-350M: **51.3** / MobileLLM-LS-350M: **52.1**
    - これまでの 350M〜430M 帯のベスト RWKV-430M は 47.0。**+4 ポイント以上**勝利。
- **TQA（クイズに答えるテスト）の F1 スコア（1-shot）**
  - MobileLLM-125M: 13.9（OPT-125M は 8.7）
  - MobileLLM-350M: 22.0（OPT-350M は 11.0）→ **約 +10 ポイント**の差。
- **AlpacaEval（おしゃべりベンチマーク）の勝率**
  - MobileLLM-LS-350M: **48.20%**（text-davinci-001 という GPT-3 ベースのモデルを相手に）。
  - text-davinci-001 が自分自身と戦った時の勝率が 50% なので、これにほぼ並んでいる。サイズが何倍もある Falcon-1.3B（30.38%）や OPT-1.3B（38.84%）を上回った。
- **API 呼び出し（スマホでアプリを呼ぶタスク）の正解率**
  - MobileLLM-350M: 意図一致 **65.3**、構造一致 **48.8** で、なんと **LLaMA-v2 7B**（意図 62.8、構造 50.9）と同等レベル。サイズは約 1/20 なのに。
- **iPhone 13 で実測した動作速度**
  - MobileLLM-LS は MobileLLM と比べて、ロード時間 +2.2%、実行時間 +2.6% で済む（ほぼ同じ）。
  - 一方、層をただ 2 倍にしただけのモデル（重み共有なし）は、ロード時間 +143%、実行時間 +86% と大幅に重くなる。**layer sharing の恩恵が実機で証明された**。
- **量子化（後述）にも強い**: W8A8 という圧縮をしても精度低下は 0.5 点未満。
- **著者の挙げる貢献（contribution）**:
  1. scaling law に反して「**深さ > 幅**」を 19 個のモデルで実証。
  2. embedding sharing と GQA を小型 AI の文脈で再評価して採用。
  3. **immediate block-wise layer sharing** という、追加パラメータゼロ・遅延ほぼゼロの精度向上テクを提案。
  4. 125M / 350M で当時の最強記録（SOTA）を更新。
  5. 同じレシピが 600M / 1B / 1.5B にもスケールすることを確認（付録）。
  6. 事前学習コードを GitHub で公開。

## キーワード辞典

専門用語の出現順。各 1 行で噛み砕いた説明。

- **LLM (Large Language Model)** … 大きな言語AI。たくさんの文章を読んで「次に来る単語」を予想することで賢くなった AI。
- **パラメータ** … AI が学習で覚えた「数」。多いほど物覚えは良いがメモリと電気を食う。
- **sub-billion** … 10 億未満。ここでは 1B = 10 億パラメータより小さい AI のこと。
- **DRAM** … スマホの「主メモリ」。アプリが動くために一時的に置かれる広い部屋。
- **SRAM** … CPU/GPU のすぐ隣にある「超高速だけど超せまい」メモリ。出し入れが速いぶん、たくさんは入らない。
- **scaling law** … 「AI の賢さはほぼ『データ量・パラメータ数・計算量』で決まる」という有名な経験則。Kaplan ら (2020) の論文。
- **FFN (Feed-Forward Network)** … AI の中の部品の 1 つ。各単語ぶんの数の並びを、ごちゃっと変換して情報を混ぜる小さな計算機。
- **SwiGLU** … FFN の中で使う計算の一種。普通の ReLU（マイナスを 0 にする関数）より、なめらかで性能が良い。Dauphin+2017 由来。
- **Transformer** … 今の AI のほぼ全部に使われている基本構造。Attention と FFN を交互に重ねたもの。
- **layer / 層 / ブロック** … Transformer の「1 段」のこと。1 つの Attention + 1 つの FFN がワンセット。
- **embedding** … 単語を「数の並び」に変えたもの。「犬」「猫」を 512 個ぐらいの数で表すと、似た意味の単語は似た数になる。
- **embedding sharing** … 入口（単語 → 数）と出口（数 → 単語）で同じ表を共有してパラメータを節約するテクニック。
- **Attention** … 「文の中のどの単語に注目するか」を計算する仕組み。AI の理解力の心臓部。
- **query / key / value** … Attention で使う 3 つの数の並び。query は質問、key は看板、value は中身（本文の式 (1) 参照）。
- **head（attention head）** … Attention を「複数の視点」で並行に行うための分け方。例えば head が 9 個なら 9 通りの注目を同時にやる。
- **Grouped Query Attention (GQA)** … query の数より少ない key/value で済ませる省エネ Attention。
- **KV cache** … 文を 1 単語ずつ生成する時、過去に計算した key/value を覚えておくメモリ。GQA はこれを小さくできる。
- **layer sharing / weight sharing** … 同じ重み（覚えた数）を、別の層でも使い回すテクニック。サイズを増やさず層を増やせる。
- **immediate block-wise sharing** … 隣り合うブロックで重みを共有する方法。SRAM に置いたまま 2 回計算できるので速い。
- **softmax** … たくさんの数を「合計が 1 の比率」に整え直す関数。大きい数ほど大きな比率になる。
- **zero-shot** … 「お手本を見せずに、いきなり本番問題を解かせる」評価方法。
- **few-shot / 1-shot** … 「お手本を 1 つだけ見せてから解かせる」評価方法。
- **fine-tuning** … 既に学習済みの AI に、特定の課題（チャット、API 呼びだしなど）専用の追加学習をして調整すること。
- **PTQ (Post-Training Quantization)** … 学習後に重みを「ざっくりした数（8bit など）」に丸めて軽くする技術。W8A8 = 重み 8bit、活性 8bit。
- **Knowledge Distillation (KD)** … 大きな先生 AI のふるまいを、小さな生徒 AI に真似させて学ばせる手法。論文では「思ったほど効かない」と negative result。
- **A100** … NVIDIA の高性能 GPU。研究用に 32 台使った、と書かれている。
- **token** … AI が文を扱う最小単位。だいたい「単語の一部」くらい。
- **AlpacaEval / MT-Bench** … チャット AI の良し悪しを、別の強い AI（GPT-4 など）に審査員になってもらって測るベンチマーク。
- **TQA (TriviaQA)** … 雑学クイズに答えるベンチマーク。
- **RACE** … 中国の中高生向け英語読解問題を使った読解力テスト。
- **ARC / BoolQ / PIQA / SIQA / HellaSwag / OBQA / WinoGrande** … 常識・推論を測る zero-shot ベンチマーク群。
- **ExecuTorch / MPS** … PyTorch をスマホ上で動かすための仕組み。Apple の GPU を使うバックエンドが MPS。

## ちょっと深掘り（中学生は飛ばして OK）

- **「深さ＞幅」の正体**: 同じパラメータ数なら、層を増やす方が「抽象的な概念」を組み立てやすい、と著者は説明している。例えば「いぬ → 動物 → 生き物 → 単語」というふうに段階的に高次の概念へまとめあげる作業は、層が多いほど得意。論文では 125M で 30〜42 層、350M で 32 層をすすめている。
- **layer sharing の 3 戦略**: Figure 5 で 3 つの方法（即時 block-wise / 全体繰り返し / 逆順）を比較していて、精度だけ見るとごく僅差で「全体繰り返し」が勝つ。でも論文は「SRAM に 1 ブロックしか乗らないので、即時 block-wise の方がメモリ転送が要らず速い」という理由で実機向けにこれを採用。精度差は seed の揺らぎと近い大きさで、論文では細かい有意性検定は提示されていない。
- **head 数の話のねじれ**: §2.2.4 では「query 16 / kv 4 がベスト」と書かれているのに、最終モデルは MobileLLM-125M で query 9 / kv 3、350M で query 15 / kv 5（付録の Table から）。これは、最終モデルでは embedding 次元を増やすために head 数を再調整した結果。head_dim はどちらも 64。
- **学習設定**: 32 × A100 GPU、各 GPU バッチ 32、Adam、weight decay 0.1、学習率 2e-3 から cosine で減衰。試作は 0.25T tokens（120k iter）、本番は **1T tokens（480k iter）**。ただし、本文中には「どんなデータセットを使ったか」は明示されていない。
- **量子化との相性**: W8A8 PTQ で精度低下が 0.5 ポイント未満。layer sharing と組み合わせても問題なし。
- **KD の失敗談（負の結果）**: LLaMA-v2 7B を先生にして KD を試したが、学習時間が 2.6〜3.2 倍になっただけで精度はほぼ同じ or わずかに悪化。小型 LLM の蒸留は割に合わない、という証拠データ。これは付録 G に書かれている。
- **scaling と限界**: 著者は 1.5B まで同じレシピでスケールすることを付録で示しているが、grid search 自体は 125M/350M 範囲。「もっと大きいモデルでも同じか？」は extrapolation（外挿）にとどまる。
- **比較の公平性の弱点（TeX 中には明示されていない論点）**: OPT、Pythia、GPT-Neo はそれぞれ異なるコーパスで学習されており、本論文の 1T tokens 学習との「データ差」と「アーキテクチャ差」を厳密に切り分けるのは難しい。注意して読むべきポイント。
