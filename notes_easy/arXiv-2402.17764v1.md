# The Era of 1-bit LLMs: All Large Language Models are in 1.58 Bits（1.58 ビット時代の大規模言語モデル）

- arXiv: https://arxiv.org/abs/2402.17764
- 一次ソース: ../papers/arXiv-2402.17764v1/
- 正規ノート: ../notes/arXiv-2402.17764v1.md

---

## 一言で言うと

巨大な AI のおもみを「-1・0・+1 の 3 種類だけ」にしても、ふつうの AI と同じ賢さで動かせる、という発見。

## どんな問題を解こうとしてるの？

- 最近の AI（**LLM**＝Large Language Model、ChatGPT のような大きな言語 AI）は賢いけど、動かすときに「電気をいっぱい食べる」「メモリをいっぱい使う」「動きがおそい」という困りごとがある。
- たとえば 700 億個（70B）のおもみを持つ AI を動かすには、80GB のメモリを積んだ高性能 GPU が 2 枚いる。これは一般の人がパソコンで動かせるレベルではない。
- これまでの工夫は「最初は普通サイズで学習させて、あとから 4 ビットに削る（**post-training quantization** = 後で量子化する＝小さい数値で表しなおすこと）」というやり方だった。でもこれは「あとから無理やり詰めなおす」ので、本当は損をしている。
- まったく別のやり方として「最初から 1 ビット（=0 か 1 の 2 種類）で学習する」**BitNet** という研究があったけど、2 種類だけだと表現の幅がせまくて、賢さがちょっと足りなかった。

## どうやって解いたの？

### コアアイデア

AI の中の「おもみ（重要度を表す数）」を、ふつうは小数点いっぱいの数（例: 0.0314, -1.2701, ...）で持つのを、たった **3 種類「-1, 0, +1」だけ**にしちゃう。これだけ。

例え話: 料理のレシピで「砂糖 3.14159 グラム、塩 0.273 グラム…」と細かく書く代わりに、「砂糖：足す（+1）／塩：なし（0）／こしょう：減らす（-1）」という 3 段階だけのレシピにする感じ。情報は減るけど、その分計算もメモリもラクになる。しかも「0（=使わない）」というえらび方ができるので、「いらない材料は本当に使わない」とハッキリ言える。これが今回 0 を増やした最大のミソ。

### 仕組み

- step1: モデルの中の「ふつうの掛け算する部品（`nn.Linear`）」を、ぜんぶ「-1, 0, +1 専用の部品（`BitLinear`）」にとりかえる。
- step2: おもみを -1/0/+1 にぎゅっと押しこむときは、「**absmean**（おもみの絶対値の平均）」で全体の大きさをそろえてから、いちばん近い -1, 0, +1 に丸める。
- step3: 文字を数字にしたあとの「活性（activation = AI の中をながれる中間の数）」は、ちょっと余裕をもたせて 8 ビット（=256 段階の整数）で表す。
- step4: 学習は最初から（=スクラッチから）この 3 値で行う。「あとから 1.58 ビットに減らす」のではなく、「最初から 1.58 ビットで育てる」のがポイント。
- step5: 全体の骨組みは、有名な **LLaMA**（Meta が出した有名な LLM）の作りをまねする（RMSNorm, SwiGLU, RoPE などの部品を採用）。こうすると、世の中の人がすでに作った道具（HuggingFace, vLLM, llama.cpp）にそのまま乗せられる。

### 主要な数式

#### 数式 1: おもみを -1, 0, +1 に丸めるルール（absmean 量子化）

$$ \widetilde{W} = \mathrm{RoundClip}\!\left(\frac{W}{\gamma + \epsilon},\ -1,\ 1\right) $$

$$ \mathrm{RoundClip}(x, a, b) = \max(a,\ \min(b,\ \mathrm{round}(x))) $$

$$ \gamma = \frac{1}{n m} \sum_{ij} |W_{ij}| $$

**この式が言ってること**: まず、おもみ全部の「絶対値の平均」($\gamma$) を計算する。次に、それぞれのおもみをその平均でわって「だいたい -1 〜 +1 の範囲」におさめる。最後に、いちばん近い整数（-1, 0, +1 のどれか）に丸める。

**記号の意味**:
- $W$ … もとのおもみの表（たくさんの数が並んだもの。$W_{ij}$ は $i$ 行目・$j$ 列目の数）
- $\widetilde{W}$ … 3 値に丸めたあとのおもみ
- $\gamma$（ガンマ）… おもみ全部の絶対値（マイナスをプラスに直した値）の平均
- $n, m$ … おもみの表のたて・よこの大きさ（個数を数えるため）
- $|W_{ij}|$ … おもみのマイナスを取りはらった値（絶対値）
- $\epsilon$（イプシロン）… 「ゼロでわらないため」の超ちっちゃい数（保険）
- $\mathrm{round}(x)$ … いちばん近い整数に丸める操作（例: 0.7 → 1, -0.3 → 0）
- $\mathrm{RoundClip}(x, a, b)$ … 丸めたあと、$a$ より小さければ $a$ に、$b$ より大きければ $b$ に押しこめる関数（ここでは $a=-1, b=+1$）

**身近な例え**: クラスの全員のテストの点数を「だいたい平均くらいの大きさ」で割って、3 段階の評価（マイナス／ふつう／プラス）にする感じ。たとえば点数を「クラス平均」でわって、「平均よりかなり下→ -1」「平均くらい→ 0」「平均よりかなり上→ +1」と振り分ける。生の点数（87 点とか 42 点とか）を捨てて、評価記号だけ残すイメージ。

#### 数式 2: 1 個のおもみが持つ情報量は何ビット？

$$ \log_2 3 \approx 1.58 $$

**この式が言ってること**: 選べる種類が 3 つあるとき、それを表すのに必要な「ビット数」はだいたい 1.58 ビットだよ、という計算。だからこの論文のタイトルが「1.58 ビット」になっている。

**記号の意味**:
- $\log_2 3$ … 「2 を何乗したら 3 になる？」という問いの答え（ここは中学生にはむずかしい関数だけど、要するに「3 通りを表すのに必要な情報量」を測るものさし）
- $3$ … えらべる種類の数（-1, 0, +1 の 3 種類）
- $\approx$ … 「だいたい等しい」という記号

**身近な例え**: コインを 1 回投げると裏か表の 2 通り（=1 ビット）。コインを 2 回投げると 2×2=4 通り（=2 ビット）。じゃあ「3 通り」を表すには？ 1 ビットだと足りない、2 ビットだとあまる、その間のだいたい 1.58 ビット。ちょうど 1 ビット（=2 通り）と 2 ビット（=4 通り）のあいだ。

### なぜ速く・軽くなるの？（しくみの直感）

ふつうの AI は「3.14 × 2.71 = ?」みたいに、小数の **掛け算** をめちゃくちゃたくさんやっている。掛け算は電気をたくさん食う。

ところが、おもみが -1, 0, +1 だけなら、「3.14 × (-1) = -3.14」「3.14 × 0 = 0」「3.14 × (+1) = 3.14」と、**掛け算がぜんぶ「そのままコピー or 符号反転 or 捨てる」だけ** になる。要するに掛け算が消えて、足し算だけが残る。足し算は掛け算よりずっと省エネ。

例え話: ぜんぶのおこづかい計算が「+1000 円もらう／-500 円つかう／0 円（なし）」だけになる感じ。電卓いらず、暗算でOK。

## 何がすごいの？

論文の数字を、TeX からそのまま引っぱってきたよ。

- **賢さ（PPL: Perplexity = 次の単語をどれくらい当てられるかの指標。低いほど良い）**
  - 700M（=7 億パラメータ）: LLaMA 12.33 vs BitNet b1.58 12.87 → まだ少し負けている
  - 1.3B（=13 億）: 11.25 vs 11.29 → ほぼ並ぶ
  - 3B（=30 億）: 10.04 vs **9.91** → ★初めて LLaMA を上回る★
  - 3.9B b1.58: **9.62** → さらに上
- **賢さ（7 タスクの平均正答率、ARCe/ARCc/HS/BQ/OQ/PQ/WGe のゼロショット評価）**
  - 3B: LLaMA 49.7 vs b1.58 **50.2**（上回る）
  - 3.9B b1.58: **51.2**（LLaMA 3B 49.7 をしっかり上回る）
- **メモリ・速度（3B モデル比較）**
  - メモリ: 7.89 GB → 2.22 GB（**3.55 倍少ない**）
  - 1 トークン出すまでの時間: 5.07 ms → 1.87 ms（**2.71 倍はやい**）
- **大きいモデルほどメリット増（70B モデル）**
  - 速度: LLaMA の **4.1 倍はやい**
  - 同時にあつかえる文（Max Batch Size）: 16 → **176**（11 倍）
  - 1 秒に出せるトークン数（Throughput）: 333 → **2977**（**8.9 倍**）
- **電気の使用量**
  - 7nm チップでの行列の掛け算にかかる電気: **71.4 倍少ない**（INT8 加算中心 vs FP16 加算＋掛け算）
- **2 兆トークンで学習したときの比較**
  - StableLM-3B（既存のすごい 3B モデル）vs BitNet b1.58 3B（2T トークン）、5 タスク平均: 73.22 → **74.34**。全タスクで勝った。
- **著者が「貢献」として強く主張していること（abstract と §1 から）**:
  - (1) ternary（3 値）のおもみで 1.58 ビット LLM を実現し、3B 規模で FP16 と賢さが並ぶことを示した
  - (2) メモリ・速度・スループット・電気のぜんぶで得をする（Pareto 改善）
  - (3) 「0 を入れたこと」が表現力（feature filtering、特徴をいらないと選びとる力）を増やす鍵だと指摘
  - (4) 1 ビット LLM 用の専用ハードウェアを設計しようと業界に呼びかけ

## キーワード辞典

本文に出てきた専門用語を、出てきた順にやさしく説明。

- **LLM (Large Language Model)** … 大規模言語モデル。たくさんの文章で学習した、ChatGPT みたいな AI。
- **量子化（quantization）** … おもみや数を「ざっくりした少ない種類」で表しなおすこと。たとえば 0.387 を「0」と書く、みたいな。
- **post-training quantization** … 最初は普通サイズで学習しおえてから、あとで小さい数値表現に詰めなおすやり方。
- **1-bit / 1.58-bit** … おもみを 2 通り（1 ビット）or 3 通り（1.58 ビット）の超少ない種類で表すこと。
- **BitNet** … この論文の前身。おもみを {-1, +1} の 2 値にしたモデル。
- **ternary（テナリー）** … 3 値（ここでは -1, 0, +1）のこと。
- **重み・パラメータ（weight, parameter）** … AI が学習で覚えた「どの情報をどれだけ大事にするか」を表す数。
- **行列の掛け算（matrix multiplication）** … たくさんの数の表どうしを掛け合わせる計算。AI の中で 1 番たくさん出てくる計算。
- **PPL (Perplexity)** … 次に来る単語をどれくらい当てられるかの指標。小さいほど賢い。
- **ゼロショット（zero-shot）** … その問題の例を一つも見せずに、いきなり解かせる評価のしかた。
- **ARC-Easy / ARC-Challenge / Hellaswag / Winogrande / PIQA / OpenbookQA / BoolQ / SciQ / LAMBADA** … AI の常識・推理力をはかる試験のセット。
- **WikiText2 / C4** … 文章のかたまり（学習や評価に使うテキストデータセット）。
- **RedPajama** … 学習に使った大きな文章データセット（100 億 〜 2 兆語）。
- **Transformer** … いまの LLM のほとんどが採用している計算のしくみ。文章を一気に見て関係を計算する仕組み。
- **LLaMA** … Meta（旧 Facebook）が出した代表的な LLM。多くの研究の標準になっている。
- **RMSNorm / SwiGLU / RoPE** … LLaMA で使われている内部の細かい部品。BitNet b1.58 もそのままマネしている。
- **bias（バイアス）** … 計算の最後に足す調整用の数。今回は使わない（=削除）。
- **HuggingFace / vLLM / llama.cpp** … みんなが使っている、LLM 用の有名な道具・ライブラリ。
- **絶対値（absolute value）** … マイナスをとっぱらった大きさのこと。例: $|-3|=3$。
- **GPU / A100** … AI 計算用の高性能チップ。A100 はその中でも代表的なやつ（80GB 版が登場）。
- **FasterTransformer / Ladder** … 速さ測定に使った既存の高速化ソフト。Ladder の「2 ビットカーネル」を借りて動かしている。
- **FP16 / BF16 / INT8** … 数の表しかた。FP16/BF16 は 16 ビット小数、INT8 は 8 ビット整数。ビット数が少ないほど省メモリで省エネ。
- **スループット（throughput）** … 1 秒間にどれだけたくさん文字を生成できるか。
- **バッチサイズ（batch size）** … 同時に何個の文を並列で処理できるか。
- **パイプライン並列（pipeline parallelism, GPipe）** … 大きいモデルを複数の GPU に分けて流れ作業で計算するやり方。
- **MoE (Mixture-of-Experts)** … 「専門家ネットワークを複数用意して使い分ける」タイプの LLM。
- **KV cache** … 長い文を処理するときに、前に計算した中間結果を覚えておく仕組み。
- **エッジ・モバイル** … スマホや小さな機器のこと。
- **CPU / NPU / LPU** … 計算用チップの種類。LPU は Groq 社が出した LLM 専用チップ。
- **Pareto 改善（Pareto solution）** … 「どの面でも損していなくて、どこかが得している」状態のこと。

## ちょっと深掘り（中学生は飛ばして OK）

- **なぜ 3B から並ぶ？** 700M や 1.3B では b1.58 はわずかに負ける（PPL も平均正答率も）。3B から FP16 と肩を並べ、3.9B では LLaMA 3B より上に行く。論文は「モデルが大きくなるほど 1.58 ビットでも表現力が足りるようになる」と解釈している（直感: 1 つ 1 つの値は粗くても、たくさん組み合わせれば滑らかさが出る）。
- **「0 を入れた」ことの意味**: 前身の BitNet は {-1, +1} の 2 値だったので、おもみは必ず「足す」か「引く」かのどちらか。0 を加えると「使わない（=この特徴は無視）」という選択肢ができる。これを論文では **feature filtering**（特徴の取捨選択）と呼んでいて、賢さを引き上げる鍵だと主張している。
- **活性化（activation）の量子化**: 元の BitNet と違って、非線形関数の手前で活性を $[0, Q_b]$ に押しこむ操作（zero-point quantization）はやめて、シンプルに $[-Q_b, Q_b]$ に正負対称で押しこむ。実装も理屈もシンプル、性能への影響もごくわずか、と書かれている。
- **新しいスケーリング則の主張**: 13B b1.58 ≈ 3B FP16, 30B b1.58 ≈ 7B FP16, 70B b1.58 ≈ 13B FP16。ただし論文本文に書かれているとおり「latency, memory usage, energy consumption」での等価であって、賢さの等価ではない点に注意（同じ計算コストで b1.58 のほうがたくさんパラメータを積めて結果として賢い、という主張）。
- **限界・著者の留保**:
  - 速度・メモリは「2 ビット用カーネル」での測定であり、1.58 ビット専用に最適化すればもっと速くなりうる（=現状の数字は控えめ）。
  - エネルギーの 71.4 倍は「行列の掛け算の計算部分だけ」。DRAM⇔SRAM のデータ移動や、softmax・LayerNorm・embedding の消費は含んでいない。
  - 活性は 8 ビットだが、将来は 4 ビット以下にしたい（future work）。
- **比較ベースラインについての注意**: 比較対象は「著者ら自身が RedPajama 100B トークンで学習した FP16 LLaMA」。公開されている LLaMA-1（1〜1.4T トークン学習）ではない。なので「FP16 LLaMA と並ぶ」の意味はやや限定的。
- **PTQ（GPTQ/AWQ など）との直接比較は本文にない**。intro で「PTQ は sub-optimal」と書いているだけで、表での数字比較は登場しない。
- **学習側の議論はうすい**: QAT（quantization-aware training = 量子化を意識した学習）は不安定になりやすいが、本文には学習の安定性・収束曲線・ハイパラ等の記述はない（短い技術レポート寄りの書きぶり）。
