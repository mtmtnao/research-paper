# Self-Refine: Iterative Refinement with Self-Feedback（自分の答えを自分でダメ出しして、自分で直していく方法）

- arXiv: https://arxiv.org/abs/2303.17651
- 一次ソース: ../papers/arXiv-2303.17651v2/
- 正規ノート: ../notes/arXiv-2303.17651v2.md

## 一言で言うと

同じ AI に「書く → ダメ出し → 直す」を何回もやらせるだけで、追加の勉強なしに答えがどんどん良くなる、という研究。

## どんな問題を解こうとしてるの？

- 大きな AI（LLM ＝ Large Language Model ＝ 大量の文章を学習させて、文章を作れるようになった AI のこと。ChatGPT などが代表）に「いい感じの返事」や「読みやすいプログラム」を 1 回でお願いすると、けっこう惜しい答えしか返ってこない。
  - たとえば友だちに「データを送って」とメールを書かせると、AI はぶっきらぼうに「データを早く送って」と書いてしまう。本当はもう少し丁寧に書きたい。
  - 「このプログラムをもっと速く書き直して」とお願いしても、最初は遅いままのコードが返ってくることが多い。
- 今までこれを直そうとすると、
  - **追加の専用 AI を学習させる**（＝もう一つ別の AI を作って「直し屋さん」にする）か、
  - **人間が大量に「これがダメ、こう直して」と教えてあげる**
  必要があって、お金も時間もかかった。
- 著者たちは「人間って、自分で書いた文章を読み返して『あ、この言い方失礼かも』と気づいて自分で直すよね？ AI にも同じことをやらせれば、追加の勉強なしで答えがよくなるんじゃない？」と考えた。

## どうやって解いたの？

### コアアイデア

**同じ 1 つの AI に、3 つの役（書く人・ダメ出しする人・直す人）を全部やらせる**。

学校で読書感想文を書くときをイメージしてほしい。

1. まず勢いで第 1 稿を書く（書く人）。
2. 書いた紙を読み返して「ここの言い方カタくない？」「最後の段落、結局何が言いたいかわからない」と自分でツッコミを入れる（ダメ出しする人）。
3. そのツッコミを見ながら、もう一回書き直す（直す人）。
4. 「もう直すところないな」と思うまで 2 と 3 を繰り返す。

この 1 〜 4 を、人間が間に入らずに、ぜんぶ 1 つの AI にやらせる。AI への指示文（プロンプト）を 3 種類用意するだけで動く。新しく AI を勉強させ直す必要はない。

### 仕組み

- step1: **最初の答えを作る**。入力 $x$（たとえば「このプログラムを速くして」）と、お手本付きの指示文 $p_{gen}$ を AI $\mathcal{M}$ に渡し、第 1 稿 $y_0$ を作らせる。
  - 「お手本付き」とは、AI に「こんな入力にはこんな出力を返してね」という例を 2〜3 個見せてからやらせる方式のこと（few-shot prompting と呼ぶ）。
- step2: **ダメ出しを作る**。同じ AI に、別の指示文 $p_{fb}$ といっしょに「さっきの答え $y_t$」を渡して、自分で自分の答えに対するダメ出し $fb_t$ を書かせる。
  - このとき AI には「**具体的に**（どこがダメか場所を指す）、**実行可能に**（どう直すかを書く）」とお手本で教えておく。「もっと効率よく」みたいなふわっとした感想じゃダメで、「この for ループは遅いから $n(n+1)/2$ の公式に書き換えよう」みたいな具体性が必要。
- step3: **直す**。同じ AI に、また別の指示文 $p_{refine}$ と「これまで作った答え全部 + これまでのダメ出し全部」を渡して、新しい答え $y_{t+1}$ を作らせる。
  - 過去の答えとダメ出しを全部渡すのは、「前にこういうミスをしたよね、もうしないでね」と思い出させるため。
- step4: step2 と step3 を、決めた回数（実験では最大 4 回）まで繰り返すか、AI 自身が「もう直すところないよ」と言うまで続ける。最後に出た答えを採用する。

### 主要な数式

#### ① 最初の答えを作る

$$ y_0 = \mathcal{M}(p_{gen} \,\|\, x) $$

**この式が言ってること**: 入力 $x$ と「こうやって書いてね」というお手本付き指示 $p_{gen}$ をつなげて AI に渡すと、最初の答え $y_0$ が出てくるよ、と言っている。

**記号の意味**:
- $y_0$ … 最初の答え（draft 第 1 稿）
- $\mathcal{M}$ … 大きな AI 本体（ChatGPT とか GPT-4 とか）
- $p_{gen}$ … 「こういう入力にはこういう出力を返してね」というお手本を並べた指示文
- $x$ … 解いてほしい今回の問題（例：「このプログラムを速くして」）
- $\|$ … 2 つの文章を「つなげる」記号。糊（のり）でくっつけるイメージ

**身近な例え**: 料理。$x$ が「今日の夕ご飯はカレーで」というリクエスト、$p_{gen}$ が「お母さんが渡してくれたレシピ集（過去の作例）」、$\mathcal{M}$ がコックさん。レシピ集を見ながら作ったのが最初のカレー $y_0$。

#### ② 自分でダメ出しする

$$ fb_t = \mathcal{M}(p_{fb} \,\|\, x \,\|\, y_t) $$

**この式が言ってること**: 同じ AI に、ダメ出し用の指示 $p_{fb}$ と元の問題 $x$、今ある答え $y_t$ をつなげて渡すと、ダメ出し文 $fb_t$ が返ってくる。

**記号の意味**:
- $fb_t$ … $t$ 回目のダメ出し（feedback の略。「ここがこうダメ、こう直して」という文）
- $\mathcal{M}$ … 同じ AI（さっきと同じ人がレビュアーもやる）
- $p_{fb}$ … 「こういう悪い答えにはこういうダメ出しをしてね」というお手本付きの指示
- $x$ … 元のお題
- $y_t$ … 今あるバージョンの答え（$t$ が反復回数）
- $\|$ … つなげる記号

**身近な例え**: テストの自己採点。$y_t$ が自分の答案、$p_{fb}$ が「○か×か、なぜ間違ったかをこう書いてね」という採点ルール、$x$ が問題文。同じ自分が答案を書いて、同じ自分が採点する。コツは「3 問目は計算ミス」と具体的に書くこと。「全体的にイマイチ」みたいなコメントだと直しようがない。

#### ③ ダメ出しを見て直す（履歴つき）

$$ y_{t+1} = \mathcal{M}(p_{refine} \,\|\, x \,\|\, y_0 \,\|\, fb_0 \,\|\, \cdots \,\|\, y_t \,\|\, fb_t) $$

**この式が言ってること**: 同じ AI に、直し方の指示 $p_{refine}$、元のお題 $x$、これまでの答え全部 $y_0, \dots, y_t$、これまでのダメ出し全部 $fb_0, \dots, fb_t$ を全部つなげて渡すと、新しい答え $y_{t+1}$ が返ってくる。「過去のミスを覚えてから直す」のがポイント。

**記号の意味**:
- $y_{t+1}$ … 次のバージョンの答え
- $p_{refine}$ … 「ダメ出しを見て、こう直してね」のお手本付き指示
- $y_0, y_1, \dots, y_t$ … これまでに作った答え全部
- $fb_0, fb_1, \dots, fb_t$ … これまでに自分で書いたダメ出し全部
- $\mathcal{M}$, $x$, $\|$ … 上と同じ

**身近な例え**: 読書感想文の書き直し。前のバージョンの感想文と、自分で書いた赤ペンメモを全部机に並べて、「前は『面白かった』としか書けなかった、今度は理由を書こう」と過去の反省を踏まえて書き直す感じ。捨てずに残しておくことで、同じ失敗を繰り返さない。

## 何がすごいの？

- **7 個のバラバラな課題で試したら、ほとんど全部で良くなった**。著者が試した課題は次の 7 つ:
  - Dialogue Response Generation（対話の返事を作る）
  - Code Optimization（プログラムを速く書き直す）
  - Code Readability Improvement（プログラムを読みやすく書き直す）
  - GSM8K（小学校〜中学校レベルの算数の文章題）
  - Sentiment Reversal（文章の気持ちを逆にする：ネガティブ↔ポジティブ）
  - Acronym Generation（略語を作る。例: "Self-Refinement Network" → "SRN"）
  - CommonGen-Hard（指定された単語を全部使って自然な文を作る難しめのテスト）
- **3 種類の AI**（GPT-3.5 の text-davinci-003、ChatGPT (gpt-3.5-turbo)、GPT-4）で試した。3 × 7 = 21 マス中、**20 マスでベース（直さない 1 回勝負）より良くなった**。平均で約 20% の絶対改善。
- 代表的な GPT-4 の改善（ベース → Self-Refine 後）:
  - Dialogue Response: **25.4 → 74.6（+49.2）**
  - Sentiment Reversal: **3.8 → 36.2（+32.4）**
  - CommonGen-Hard: **15.0 → 45.0（+30.0）**
  - Code Readability: **27.4 → 56.2（+28.8）**
  - Acronym: **30.4 → 56.0（+25.6）**
  - Code Optimization: **27.3 → 36.0（+8.7）**
  - GSM8K（算数）: 92.9 → 93.1（+0.2、ほぼ変わらず）
- **ダメ出しの質がすごく大事**だと数字で証明した（Table 3）:
  - Sentiment Reversal: 具体的なダメ出し **43.2** vs ふわっとしたダメ出し **31.2** vs ダメ出しなし **0**
  - Code Optimization: **27.5** vs **26.0** vs **24.8**
  - Acronym: **56.4** vs **54.0** vs **48.0**
- **早い反復で一気に良くなる**（Fig. 4）。Constrained Generation の例だと、1 回目→2 回目で +11.3、2 回目→3 回目で +6.4、3 回目→4 回目で +3.0。だんだん伸びが小さくなる（＝1〜2 回でやめても十分得）。
- 著者が論文中で挙げている貢献（contributions）:
  1. 追加学習も外部の人間も使わずに「同じ AI でぐるぐる直す」だけで答えを良くできる仕組みを提案。
  2. 似た先行研究（PEER、Reflexion、CodeRL、Self-correction など）と 4 つの条件（教師データ不要 / 専用直し AI 不要 / 複数の観点を見る / 反復する）で比較し、4 つ全部を同時に満たすのは Self-Refine だけ、と整理した（Table 2）。
  3. **CommonGen-Hard** と **Acronym Generation** という新しい評価課題を作った。
  4. ダメ出しの質・反復回数・サンプル数・モデルサイズなどを細かく分けて分析した。

## キーワード辞典

- **LLM (Large Language Model)** … 大量の文章を学習して、文章を作ったり質問に答えたりできるようになった大きな AI。ChatGPT などが代表例。
- **GPT-3.5 / ChatGPT / GPT-4** … OpenAI 社が作った LLM の名前。GPT-4 が一番賢い。
- **Codex (code-davinci-002)** … プログラムを書くのが得意な LLM。
- **Vicuna-13B** … 小さめで弱めの LLM。実験で「Self-Refine は弱い AI だと動かないのか？」を確かめるのに使われた。
- **few-shot prompting（フューショット・プロンプティング）** … AI に何かやらせるとき、「こういう入力にはこう答えてね」という例を 2〜3 個見せてからお願いする方法。例を見せた回数の少なさ（few-shot ＝ 少数事例）から来ている名前。
- **prompt（プロンプト）** … AI への指示文。「これやって」と書いた紙のようなもの。
- **iterative（イテラティブ）** … 何度も繰り返すこと。
- **feedback（フィードバック）** … 「ここが良くない、こう直して」というダメ出し・改善提案。
- **refine（リファイン）** … 答えを直して質を上げること。
- **actionable feedback** … 「具体的に何をすれば直るか」が書いてあるダメ出し。「もっと頑張れ」じゃなく「3 問目の符号がマイナスだよ」のレベル。
- **specific feedback** … 「どこを直すべきか」が場所まで指してあるダメ出し。
- **baseline（ベースライン）** … 比較の基準になる、何もせずに 1 回だけ答えさせた結果。Self-Refine がこれにどれだけ勝てるかを見る。
- **GSM8K** … 小学校〜中学校レベルの算数の文章題を 8500 問集めたテスト。
- **CommonGen / CommonGen-Hard** … 指定単語を全部使って自然な文章を作る課題。-Hard は難しめバージョン（この論文で新規作成）。
- **supervised training（教師あり学習）** … 「これが正解だよ」と人間が答えを付けたデータで AI に勉強させる方法。Self-Refine はこれを使わない。
- **reinforcement learning（強化学習・RL）** … 「いい行動には ご褒美、悪い行動には罰」をあげて AI を訓練する方法。Self-Refine はこれも使わない。
- **temperature（テンパラチャー、温度）** … AI が答えを作るときの「ランダムさ」を調整する数字。0 だと毎回同じ答え、大きいほどバラついた答えになる。論文では 0.7。
- **best-of-k / sampling（サンプリング）** … AI に同じ問題を $k$ 回解かせて、その中で一番良いのを選ぶ作戦。Self-Refine はこれより強かった。

## ちょっと深掘り（中学生は飛ばして OK）

- **GSM8K（算数）でだけほぼ改善しなかった理由**: ChatGPT に「自分の答えにダメ出ししてね」と頼むと、**94% のケースで "everything looks good"（全部大丈夫）と返してきてしまった**。つまり、自分の計算ミスを自分で見つける力がそもそも無かった。算数のように「正解か不正解か」が客観的に決まる課題では、計算機やプログラム実行といった「外部の答え合わせ役」と組み合わせないと、自己採点はあてにならない。
- **失敗例を 35 個手でくわしく調べた結果**: 失敗の原因は、
  - 33%: ダメ出しが「場所」を間違えた
  - 61%: ダメ出しは場所は合ってるが「直し方」が間違っていた
  - 6%: 直し屋（refiner）の実装ミス
  つまり失敗の 94% は「ダメ出し」のせいで、直す側はわりとちゃんとしている。一方で **成功例の 33% は部分的に間違ったダメ出しでも直し屋がカバーしていた**。直し屋はそこそこタフ。
- **弱い AI では動かない**: Vicuna-13B でやると、「ダメ出しはこの形式で書いてね」という指示の形式すら守れなかった。それどころか、正解のダメ出しを人間が手で渡しても、それを使って直すことができなかった。つまり Self-Refine は **指示にちゃんと従える強い AI が前提**。
- **「1 本を直す」vs「何本も書いて一番良いの選ぶ」**: 同じ AI に独立に 4 個答えを書かせて一番良いのを選ぶ作戦と比べても、Self-Refine（1 本を 4 回直す）の方が人間に好まれた。
- **温度 0.7 で "greedy decoding"**: 論文には「greedy decoding を温度 0.7 で使う」と書かれているが、これは厳密には矛盾した表現（greedy = 温度 0 を意味するため）。実際にはサンプリング温度 0.7 で動かしている、と既存ノートでは整理されている。
- **過去の履歴を全部つなげて渡す**: 反復が増えるほど AI に渡す文章は長くなる。この計算コストが反復ごとに増える点について、論文中では詳しい議論はされていない（TeX に書かれていない）。
- **stop score（自己停止スコア）**: ダメ出し文の中に「もう直さなくていいかも度」を AI に数字で書かせ、それを見て打ち切る仕組みがある。が、「その自己判断がどれくらい正確か」の定量評価は本文中には見当たらない。
- **判定が GPT-4 頼り**: 多くの課題で「答えが良くなったか」を判定するのも GPT-4。生成者と判定者が同じ系列なので、ちょっと身内びいきが心配（人間 A/B 評価も一部やってはいる）。
